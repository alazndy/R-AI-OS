# R-AI-OS 2.0: Universal Agent Kernel (Aura Hardened)

## 1. Overview
R-AI-OS 2.0 transforms the system from a robust CLI/Daemon toolkit into a truly universal "Agent Operating System Kernel." It bridges multiple AI agent ecosystems (Antigravity 2.0, Codex, Claude, Gemini, OpenCode) through a hybrid, multi-protocol architecture, providing deep context, secure isolation, and asynchronous processing power.

## 2. Core Architecture: Tri-Protocol Interface
To achieve universal compatibility, R-AI-OS exposes its capabilities through three concurrent interfaces:
*   **MCP-First:** Standardized Model Context Protocol endpoints allowing compliant agents (Claude, Gemini, standard Antigravity) to access R-AI-OS as a tool server seamlessly.
*   **Daemon-First (Aura IPC / TCP):** A low-latency, bi-directional socket connection serving as the primary integration for advanced systems like Codex and Antigravity 2.0, enabling continuous context streaming and background task dispatch.
*   **CLI-First:** Traditional terminal commands serving as the lowest-common-denominator fallback for shell-scripting agents and manual user interventions.

## 3. Execution Model: Hybrid Radar-Factory Engine
The system employs a dual-mode execution strategy to handle both interactive and long-running tasks without blocking the agent:
*   **Radar Mode (Context-Aware Streaming):** A continuous background process attached to active sessions. It provides real-time "whispers" (via IPC) regarding codebase rules, security vulnerabilities (CVEs), and architectural violations as the agent types or executes commands.
*   **Factory Mode (Unified Event Bus):** An asynchronous task queue backed by SQLite. Agents offload heavy operations (e.g., massive refactoring, comprehensive test generation, or builds). The system returns a job ID immediately, allowing the agent to disconnect or focus elsewhere, reporting back via an "Inbox" or webhook when complete.

## 4. State Management: Centralized Authority + Versioned Context
R-AI-OS prevents agent collisions and protects the main codebase through a hybrid state management model:
*   **Central Authority (Lockdown):** R-AI-OS acts as the ultimate source of truth. It manages file-level and task-level locks. If Agent A (e.g., Gemini) is modifying a module, R-AI-OS blocks Agent B (e.g., Codex) from concurrent destructive edits, ensuring order in a multi-agent swarm.
*   **Versioned Context (Git Worktree Isolation):** When an agent accepts a significant task, R-AI-OS spins up a localized, temporary Git worktree. The agent operates entirely within this isolated "sandbox." Upon successful validation (tests, compliance), R-AI-OS merges the worktree back into the main branch.

## 5. Universal Capability Layer: Proxy-Store Bridge
A unified mechanism to handle diverse plugin and skill formats (Antigravity Plugins, Gemini Skills, Codex Extensions):
*   **Universal Store:** A centralized repository within R-AI-OS containing ready-to-use skills, rules, and scripts mapped to various platform requirements.
*   **Smart Proxy:** Agents interact with R-AI-OS as a proxy. The agent requests a generic tool action, and R-AI-OS translates this request into the native implementation (whether it's an internal Rust module, an external Python script, or a bridged MCP server), abstracting away the underlying complexity from the agent.

## 6. Implementation Strategy
This specification serves as the foundation for the Phase 5 (Agent Swarm Mesh) and Phase 6 (Multi-Protocol Bridge) implementation plans. The focus will initially be on establishing the Tri-Protocol Interface and Worktree Isolation mechanisms.
