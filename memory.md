## Technical Decisions
- **Architecture**: Modular kernel — `src/security/`, `src/kernel/`, `src/control_plane.rs`, `src/swarm/`, `src/cortex/`
- **Database**: SQLite via rusqlite (bundled). `cp_*` schema is the sole source of truth. Legacy tables are read-only cache.
- **Control Plane**: `cp_daemon_snapshot()` = single-call operational view. `cp_scheduler_list_ready()` = canonical task scheduler.
- **Security Kernel**: 4 layers — Filesystem Jail → Policy Manager → Verify Chain (hash-chained audit ledger) → Egress Filter
- **Policy**: `raios-policy.toml` is the single security config source. `confirm` = fail-closed (deny) in daemon/stdio mode.
- **Egress**: Filtered at MCP tool hook level (not OS network layer — scoped to what raios controls).
- **Hash Chain**: Tamper *detection*, not prevention — correctly classified as forensic tool.
- **IPC**: VS Code extension proxies requests to daemon (port 42071) via Bearer token — token never exposed to WebView.
- **Critical Notes**: New work must read/write through `cp_*` functions in `src/db.rs`. Never write directly to legacy tables.

- Situation: designing an IPC transport meant to be reachable from inside a sandboxed coding agent's shell commands (Codex, Copilot, Cursor). Signal: unit tests pass because they call the transport from the unsandboxed test process, not from a real sandboxed child. Response: before declaring a socket/pipe-based channel "sandboxed-agent-safe," live-test it from an actual `codex exec --sandbox workspace-write` (or equivalent) invocation, not just an in-process test — Landlock/seatbelt sandboxes can deny `connect()` on a Unix domain socket even when its containing directory is on the allow-list, because the restriction is on the syscall/socket-family, not the path. Loopback TCP is reachable through the *same* sandbox but only when its network-access config flag is explicitly enabled (not the default) — switching transport families changes what's *possible*, not what's true out of the box; re-verify against the sandbox's actual default config, not just its most permissive one, before calling a fix complete. Expected payoff: catches a whole class of "works in tests, fails for the real sandboxed caller it was built for" bugs that no amount of same-process testing can surface, and avoids declaring victory on a fix that only works with a non-default flag the caller doesn't know to pass.
- Situation: exposing an existing privileged domain service through a local CLI. Signal: the service already has a typed canonical dispatcher with ownership, feature-gate, idempotency, audit, and secret checks. Response: parse a bounded typed local file, reject human-only variants before opening the mutation path, then delegate instead of adding a parallel write path. Expected payoff: transport parity without authority drift.
- Situation: extending R-AI-OS memory with raw agent transcripts. Signal: workspace.db is already 2.8 GB against its 500 MB budget and the curated L0-L3 memory already exists. Response: keep any rebuildable transcript-recall index in a separate owner-only cache, expose it as read-only recall, and preserve workspace.db for curated facts and control-plane state. Expected payoff: retroactive recall without worsening DB bloat or authority confusion.
- Situation: an MCP tool manifest changes. Signal: a rebuilt binary reports a manifest hash mismatch. Response: rebuild both CLI and daemon binaries, update every installed binary path used by PATH and systemd, then run pin-reset, restart the daemon, and verify a live tool call. Expected payoff: new agent tools become available without bypassing supply-chain pinning.

Expected payoff: incomplete work is visible and recoverable without relying on UI behavior.
- Situation: an intake workflow evolves after early sessions may already exist. Signal: a new required prompt set is introduced. Response: version prompts with stable keys, seed them transactionally for new sessions, and repair missing prompts on session resumption without overwriting answers. Expected payoff: consistent discovery data without invalidating earlier local work.
- Situation: adding a local TUI workflow for a privileged product lifecycle feature. Signal: the UI needs to submit commands but public exposure is not approved. Response: use an isolated typed command contract over existing authenticated IPC, derive locality at the daemon, load the feature gate only from trusted config, and let the server enforce authorization and validation. Expected payoff: usable local flow without duplicating security policy in the UI or accidentally opening an API.
- Situation: adding a product-lifecycle write flow before public transport exists. Signal: data crosses workspace, product, intake, and charter boundaries. Response: keep the command surface disabled by default, derive actor identity from the transport, bind every write to stored ownership, hash the full payload for idempotency, and audit in the same transaction. Expected payoff: safe local iteration without accidental authority expansion.
- Situation: Adding a new control-surface summary to an evolving daemon protocol. Signal: newer UI data must coexist with older daemon snapshots and the feature must not create mutation authority. Response: add an additive default-valued snapshot field, project it from canonical read-only queries, and render it inside an existing route. Expected payoff: visible lifecycle state without daemon restart coupling, transport breakage, or accidental write paths.
- Situation: Adding a new persisted config section in a cross-surface Rust workspace. Signal: Config has direct struct literals in setup and auto-detection flows, so a new field compiles in Default but fails elsewhere. Response: locate every Config literal, preserve serde defaults for legacy files, and check all consuming surface crates. Expected payoff: prevents partial configuration migrations and runtime setup regressions.
- Situation: Designing a product-lifecycle layer on R-AI-OS. Signal: the legacy Factory is an async shell-job queue while workspace.db growth is dominated by rebuildable search indexes. Response: reserve Factory for the product domain, migrate the queue toward JobExecutor, and separate durable control state, rebuildable search cache, and content artifacts before adding lifecycle tables. Expected payoff: prevents semantic collision, unsafe execution inheritance, and unbounded canonical DB growth.
- When a new TUI field depends on a daemon restart but active agents make restart unsafe, enrich only canonical project paths inside the configured workspace with a bounded read-only local fallback; never trust remote snapshot paths for filesystem reads.
- When extending a control-plane DTO consumed by a long-running daemon, mark additive fields with serde(default) and test an older JSON payload so a freshly updated TUI remains compatible before the daemon restarts.
- When shipping cross-platform Rust changes from a Linux host, keep Windows behavior behind cfg branches and validate the full workspace in Windows CI; local GNU cross-checks can stop at a missing MinGW linker and do not prove runtime installer behavior.
- Situation: Recovering a partially implemented control-plane feature with a newly added E2E test. Signal: the test awaits a server-side accept before starting its client and the handler opens a global DB instead of the fixture. Response: remove the invalid harness, test the actual workflow against an isolated database, and add dependency injection before claiming transport-level E2E coverage. Expected payoff: prevents hangs and false confidence from tests that never exercise the intended state.
- Situation: Determining an agent's current R-AI-OS work from control-plane data. Signal: cp_agent_runs reports many sessions as running but their event streams are empty. Response: do not infer ownership or task content from session status alone; correlate sessions with active-process evidence, git diff, and a concrete handoff/task artifact before acting. Expected payoff: avoids false active-task assumptions and accidental concurrent edits.
- When using rustfmt in a Rust module tree, avoid running it on mod.rs unless skip_children is set; otherwise rustfmt can traverse child modules and create unrelated format diffs, including forbidden conflict files.
- When a generated context command is part of final verification, run git status afterward because some tools update timestamp-only files and can make a clean worktree dirty again.
- When troubleshooting timing/performance regressions, always run a benchmark with a release/optimized profile, as unoptimized debug profile math operations (like building HNSW graph indices) can be orders of magnitude slower and present false-positive regressions.
